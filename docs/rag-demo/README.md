# RAG Demo — Haystack + Qdrant

Self-hosted RAG using the project stack: **Qdrant** (vector store) + **Haystack** (pipelines, chunking, embedding) + **MCPHub** (`qdrant-mcp` for tool-calling retrieval from LobeChat).

No external API. All embeddings local via `sentence-transformers/all-MiniLM-L6-v2` (384-dim).

## Pipelines

Two Haystack pipelines, one per phase. Both rendered to PNG via `pipeline.draw(path=...)` (Mermaid).

### Indexing — `index_docs.py` → `pipeline_index.png`

```
sources/  ──►  FileTypeRouter  ──►  PyPDFToDocument
                              └──►  TextFileToDocument  ──►  DocumentJoiner
                              └──►  HTMLToDocument
                                                       ──►  DocumentSplitter (200 words, 30 overlap)
                                                       ──►  SentenceTransformersDocumentEmbedder
                                                       ──►  DocumentWriter ──►  Qdrant (collection: rag-demo)
```

Step-by-step:

| # | Component | What it does |
|---|-----------|--------------|
| 1 | `FileTypeRouter` | Inspects each path's MIME type. Routes PDF / TXT+MD / HTML to the matching converter. |
| 2a | `PyPDFToDocument` | Extracts text from PDFs page by page (uses `pypdf`). One Haystack `Document` per source. |
| 2b | `TextFileToDocument` | Reads `.md`/`.txt` raw → `Document`. |
| 2c | `HTMLToDocument` | Strips boilerplate via `trafilatura`, keeps main article text. |
| 3 | `DocumentJoiner` | Merges the three converter outputs into one stream. |
| 4 | `DocumentSplitter` | Splits by word; chunk size 200, overlap 30 → smaller `Document`s, each kept aware of source via `meta.file_path`. |
| 5 | `SentenceTransformersDocumentEmbedder` | Loads `sentence-transformers/all-MiniLM-L6-v2` locally (CPU). Adds 384-dim embedding to each chunk. Batched. |
| 6 | `DocumentWriter` | Upserts chunks into `QdrantDocumentStore(url=http://localhost:47010, index="rag-demo", embedding_dim=384, recreate_index=True)`. Recreate wipes prior runs to keep indexing idempotent. |

Outcome (last run): 8 source files → **203 chunks** in Qdrant.

### Query — `query.py` → `pipeline_query.png`

```
question  ──►  SentenceTransformersTextEmbedder  ──►  QdrantEmbeddingRetriever  ──►  top_k=4 Documents
```

| # | Component | What it does |
|---|-----------|--------------|
| 1 | `SentenceTransformersTextEmbedder` | Same model as indexing (must match dim + tokenizer). Embeds the question into 384-d vector. |
| 2 | `QdrantEmbeddingRetriever` | Cosine ANN search inside `rag-demo`. Returns top-4 `Document`s with `score` + original chunk text + `meta.file_path`. |

No generator (LLM) wired in — this demo is **retrieval-only** to keep the test deterministic and provider-agnostic. Plug a `OpenAIGenerator` / `OpenRouter` / local vLLM generator after the retriever to get a full RAG answer.

## Run

Stack must be up (`docker compose up -d`) so Qdrant is reachable on `localhost:47010`.

### Index (MCP-compatible — use this one)

```bash
uv run --with 'haystack-ai' --with 'pypdf' --with 'trafilatura' \
       --with 'fastembed' --with 'qdrant-client' \
       python docs/rag-demo/index_docs_fastembed.py
```

Output (last run): **203 chunks upserted**.

**Why this and not `index_docs.py`?** `mcp-server-qdrant` (the binary behind the `qdrant-mcp` plugin) embeds queries with **fastembed** and expects a collection that has a **named vector** called `fast-all-minilm-l6-v2`. Haystack's default `QdrantDocumentStore` writes unnamed vectors, which makes `qdrant-find` fail with:

```
Wrong input: Not existing vector name error: fast-all-minilm-l6-v2
```

`index_docs_fastembed.py` keeps the Haystack chunking pipeline (PDF / TXT+MD / HTML → join → split) but embeds with fastembed and upserts directly via `qdrant-client`, using the named vector slot and the `{document, metadata}` payload shape that `mcp-server-qdrant` reads back when rendering tool results.

`index_docs.py` is kept for reference (it's still a valid pure-Haystack pipeline if you only query through the Haystack `query.py` script).

### Query

```bash
uv run --with 'haystack-ai' --with 'qdrant-haystack' --with 'sentence-transformers' \
       python docs/rag-demo/query.py "What is retrieval-augmented generation and why does it help LLMs?"
```

### Hayhooks MCP — `http://localhost:47013/mcp`

Streamable-HTTP MCP endpoint exposing every deployed Hayhooks pipeline as an MCP tool. Already wired into MCPHub as the `haystack-mcp` server, so any MCP client (LobeChat, Claude Desktop, Cursor) can call deployed pipelines through it.

Initialize handshake (same flow as the other MCP traces in this repo):

```bash
curl -sN -X POST http://localhost:47013/mcp \
  -H "Accept: application/json, text/event-stream" \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize",
       "params":{"protocolVersion":"2024-11-05","capabilities":{},
                 "clientInfo":{"name":"trace","version":"1"}}}'
```

### MCPHub — `http://localhost:47008`

Aggregates `qdrant-mcp` + `haystack-mcp` (and others) under one MCP surface. Auth via JWT.


## LobeChat prompts to drive the RAG

`mcp-server-qdrant` is configured against the `rag-demo` collection (see `config/mcp_settings.json`, env `COLLECTION_NAME=rag-demo`). The plugin `mcphub-qdrant` exposes two tools to the model:

- `qdrant-find(query, ...)` — semantic search inside `rag-demo`. Returns top chunks with their source.
- `qdrant-store(information, metadata)` — write new content into the same collection.

Enable the **mcphub-qdrant** plugin on the agent in LobeChat, then paste the system prompt and try the user prompts below.

### LobeChat agent setup

Create a new agent in LobeChat (`+ Create Agent`) and configure it as follows. The two non-obvious bits are the plugin and the model — the system prompt does not turn into tool calls by itself, the model has to be one that supports function calling and the plugin has to be enabled on this specific agent.

| Field | Value |
|-------|-------|
| **Agent name** | `RAG Researcher` |
| **Description** | Grounds answers in the local Qdrant `rag-demo` corpus via MCP. |
| **Avatar** | any (📚 works) |
| **Model** | `google/gemma-4-E4B-it` served by local vLLM (Provider: OpenAI-compatible, BaseURL `http://vllm:8000/v1` from inside compose, or `http://localhost:47007/v1` from host browser; API key `sk-local`). Any Anthropic / OpenAI / OpenRouter model with function calling also works. |
| **Plugins** (toggle ON) | `mcphub-qdrant` (mandatory). Optionally `mcphub-fs`, `mcphub-aws-docs` for cross-source answers. |
| **Tool choice** | `auto` — let the model decide when to call `qdrant-find`. |
| **Function calling** | Enabled (default for tool-capable models). |
| **Temperature** | 0.2 — keep it factual. |
| **Top-p** | 0.9 |
| **Max history** | 8–12 messages — enough context for follow-ups, not so much that prompt grows past 16K. |
| **Opening message** | `Ask me anything about RAG, Haystack, Qdrant or MCP. I will search the local corpus before answering.` |

After saving, send the first message to verify: in the LobeChat plugin pane you should see `qdrant-mcp-qdrant-find` invoked, with the JSON arguments and chunk results expanded in the message thread.

### Role / System prompt

```
You are a research assistant grounded in a private RAG corpus stored in Qdrant
(collection "rag-demo"). The corpus contains:
  - Retrieval-augmented generation papers (Lewis 2020, Gao 2023 survey)
  - Wikipedia article on RAG
  - READMEs for Haystack, Hayhooks, Qdrant, Model Context Protocol, mcp-server-qdrant

Rules:
1. For every factual claim, FIRST call the `qdrant-find` tool with a focused
   query and read the returned chunks before answering.
2. Issue multiple `qdrant-find` calls if the question has several facets — one
   query per facet, then synthesize.
3. Quote evidence inline using markdown blockquotes and cite the source file
   from the chunk metadata, e.g. `(02_rag_survey_gao_2023.pdf)`.
4. If `qdrant-find` returns no relevant chunk, say "not in corpus" — never
   fabricate. You may still answer from general knowledge but mark it clearly
   as `[outside corpus]`.
5. End every answer with a "Sources" list of the unique files actually used.

Style: precise, technical, no filler. Prefer short paragraphs and tight bullets.
```

### Sample user prompts

Each prompt is designed to force at least one (often several) `qdrant-find`
calls and produce an answer grounded in the indexed material.

1. **Definition + history**
   ```
   Define retrieval-augmented generation in one paragraph, then summarize how
   the technique evolved between Lewis 2020 and the Gao 2023 survey. Quote at
   least one passage from each paper.
   ```

2. **Comparative architecture**
   ```
   Compare Haystack and Hayhooks: what is each, how do they relate, and when
   would I run Hayhooks instead of using the Haystack Python API directly?
   Ground every claim in the corpus.
   ```

3. **Tooling deep-dive**
   ```
   Explain what the mcp-server-qdrant project exposes as MCP tools and how a
   client invokes them. Include the exact tool names and the kind of arguments
   they accept, citing the README in the corpus.
   ```

4. **Design decision**
   ```
   I'm choosing a vector database for a RAG project with strict on-prem
   requirements. Build the case for Qdrant using only what's in the corpus —
   storage model, search modes, integrations. Flag anything the corpus does not
   answer.
   ```

5. **Multi-hop synthesis**
   ```
   Walk me through a minimal end-to-end RAG pipeline with Haystack indexing
   into Qdrant and serving over MCP. For each step name the component
   responsible, citing the relevant README chunk.
   ```

6. **Trap — out of scope**
   ```
   What is the price per million tokens of OpenAI's gpt-4.1-mini in 2026?
   ```
   Expected behavior: model calls `qdrant-find`, gets nothing relevant,
   responds "not in corpus" (and optionally answers from general knowledge
   under `[outside corpus]`).

### What a good answer looks like

For prompt 1 the model should fire something like:

```
qdrant-find({"query": "definition retrieval-augmented generation"})
qdrant-find({"query": "Lewis 2020 RAG seq2seq retriever"})
qdrant-find({"query": "Gao 2023 RAG survey naive advanced modular"})
```

…then write 2-3 paragraphs each backed by a quoted chunk and ending with:

```
Sources:
- 08_rag_wikipedia.html
- 01_rag_lewis_2020.pdf
- 02_rag_survey_gao_2023.pdf
```

### Quick sanity check from the shell

Confirm the corpus is reachable through MCP before testing in LobeChat:

```bash
TOKEN=$(curl -s -X POST http://localhost:47008/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"admin123"}' | jq -r .token)

# init MCP session
curl -sN -X POST http://localhost:47008/mcp \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Accept: application/json, text/event-stream' \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize",
       "params":{"protocolVersion":"2024-11-05","capabilities":{},
                 "clientInfo":{"name":"t","version":"1"}}}' \
  -D /tmp/h.txt -o /dev/null
SID=$(grep -i mcp-session-id /tmp/h.txt | awk '{print $2}' | tr -d '\r')

# fire qdrant-find
curl -sN -X POST http://localhost:47008/mcp \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Accept: application/json, text/event-stream' \
  -H 'Content-Type: application/json' \
  -H "mcp-session-id: $SID" \
  -d '{"jsonrpc":"2.0","id":2,"method":"tools/call",
       "params":{"name":"qdrant-mcp-qdrant-find",
                 "arguments":{"query":"what is retrieval augmented generation"}}}'
```

Should return chunks with `file_path` metadata pointing at files in `sources/`.
